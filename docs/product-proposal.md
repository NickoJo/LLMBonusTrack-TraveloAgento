# Product Proposal — TravelAgent

## 1. Обоснование идеи

Самостоятельное планирование поездки — фрагментированный процесс.
Пользователь вынужден переключаться между Skyscanner, Booking, Airbnb, Google Maps, TripAdvisor и держать всё это в голове, вручную проверяя совместимость дат, стыковок и бюджета.
Туроператоры решают эту проблему, но берут существенную наценкуи предлагают ограниченный выбор.

TravelAgent — мультиагентная система, которая ведёт пользователя через диалог, самостоятельно агрегирует данные из разных источников и собирает оптимальный travel-пакет с учётом всех ограничений пользователя.

---

## 2. Цель и метрики успеха

### Цель
Позволить пользователю получить полный план поездки (рейс + жильё + бюджет) за один диалог, не покидая интерфейс.

### Продуктовые метрики

| Метрика | Целевое значение |
|---|---|
| Plan completeness | Финальный план содержит рейс, жильё, итинерарий и бюджет — 100% |
| Budget accuracy | Итоговая стоимость не превышает заявленный бюджет более чем на 10% |
| Turns to completion | Пользователь получает готовый план за ≤ 10 сообщений |
| User correction rate | Пользователь вносит правки в план в < 30% сессий |

### Агентские метрики

| Метрика | Целевое значение |
|---|---|
| Intent extraction accuracy | IntentAgent корректно извлекает все обязательные параметры ≥ 90% |
| Search result relevance | ≥ 80% возвращённых вариантов соответствуют заданным фильтрам |
| Fallback trigger rate | Failover срабатывает корректно при 100% симулированных падений API |
| Edge case handling | Система не падает и выдаёт осмысленный ответ в 100% edge-кейсов из тест-сьюта |

### Технические метрики

| Метрика | Целевое значение |
|---|---|
| p95 latency (полный план) | ≤ 30 секунд |
| p95 latency (один агент-шаг) | ≤ 8 секунд |
| Uptime PoC-стенда | ≥ 95% в период демо |
| PII в логах | 0 инцидентов |

## 3. Сценарии использования

### Happy path
Пользователь: *«Хочу слетать в Тюмень на 5 дней в мае, нас двое, бюджет 150000 рублей»*

1. IntentAgent фиксирует: destination=Tyumen, duration=5d, month=May, travelers=2, budget=150000 currency=Rub
2. SearchAgent параллельно ищет рейсы и жильё
3. OptimizationAgent собирает лучшую комбинацию по цене + удобству
4. BudgetTrackerAgent подтверждает: пакет = 1340€, остаток 160€
5.  ItineraryAgent строит план на 5 дней
6. Пользователь получает полный план

### Edge-кейсы

| Сценарий | Ожидаемое поведение |
|---|---|
| Бюджет нереалистичен (200€ на неделю в Лондоне) | Агент честно сообщает о несоответствии и предлагает альтернативные направления или даты |
| MCP Airbnb недоступен | Failover на Booking.com, пользователю сообщается об ограниченной выборке |
| Пользователь меняет город в середине диалога | Orchestrator сбрасывает только результаты поиска, сохраняет даты и бюджет |
| Рейсов на точные даты нет | Агент предлагает ±3 дня с объяснением разницы в цене |
| Стыковка менее 50 минут | Risk-флаг, агент предупреждает и предлагает альтернативу |
| Пользователь не указал гражданство для визовых направлений | IntentAgent уточняет перед поиском |
| Запрос полностью открыт («хочу куда-нибудь») | Агент задаёт уточняющие вопросы, не падает и не выдаёт случайный результат |

## 4. Ограничения

### Технические (SLO)

| Параметр | Ограничение |
|---|---|
| Контекстное окно | История диалога ограничена ~20 сообщениями в PoC; старые сообщения суммаризируются |
| Параллельные сессии | PoC поддерживает до 5 одновременных пользователей |
| Размер результатов поиска | Каждый SearchAgent возвращает топ-5 вариантов, не более |
| Зависимость от внешних API | Все внешние интеграции — mock/sandbox; production API требуют отдельных соглашений |

### Операционные

| Параметр | Ограничение |
|---|---|
| Реальное бронирование | Out-of-scope в PoC, требует PCI DSS compliance |
| Персональные данные | Паспортные данные и платёжная информация не обрабатываются в PoC |

### Модули и интеграции

| Модуль | Роль | Интеграция |
|---|---|---|
| IntentAgent | Диалог, уточнение параметров поездки | LLM |
| SearchAgent | Параллельный поиск рейсов и жилья | MCP → mock Airbnb, Booking, Skyscanner |
| OptimizationAgent | Сборка оптимального пакета | LLM + собственная логика ранжирования |
| ItineraryAgent | Day-by-day план | LLM + RAG по travel KB |
| BudgetTrackerAgent | Отслеживание бюджета | Детерминированная логика |
| Orchestrator | State, routing, retry | Фреймворк (LangGraph / custom) |
| PII Guard | Анонимизация до LLM-вызовов | Guardrail (regex + NER) |

## 6. Data Flow и разделение LLM / детерминированная логика
```
[User message]
      │
      ▼
[IntentAgent — LLM]         Понимание естественного языка,
      │                     уточняющий диалог, извлечение
      │                     структурированного TripProfile
      ▼
[TripProfile: JSON]         Детерминировано: валидация схемы,
      │                     проверка обязательных полей
      ▼
[SearchAgent — Tool calls]  Детерминировано: параллельные
      │                     MCP-запросы, нормализация ответов,
      │                     фильтрация по бюджету/датам
      ▼
[SearchResults: JSON]
      │
      ▼
[OptimizationAgent — LLM]   LLM для оценки «удобства»
      │                     (время вылета, район жилья),
      │                     детерминировано для суммы бюджета
      ▼
[SelectedPackage: JSON]
      │
      ├──► [BudgetTracker — детерминировано]  Арифметика бюджета
      │
      ▼
[ItineraryAgent — LLM + RAG]  LLM строит план,
      │                        RAG подтягивает POI/рестораны
      ▼
[FinalPlan: JSON → Markdown]
      │
      ▼
[ReportFormatter — шаблон]  Детерминировано: рендер
                             финального ответа пользователю
```

**Правило разделения:** всё, что можно сделать детерминировано (фильтры, арифметика, валидация, форматирование) — делается без LLM. LLM вызывается только там, где нужно понимание языка, оценка субъективных критериев или генерация текста.
